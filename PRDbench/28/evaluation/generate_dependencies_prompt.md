# 测试依赖关系生成Prompt

## 任务描述
根据`detailed_test_plan.json`中的测试计划，分析测试点之间的依赖关系，生成`test_dependencies.json`文件。

## 分析原则

### 1. 数据依赖分析
- **企业信息录入**类测试点(2.1.x)是其他功能的基础数据源
- **融资诊断分析**类测试点(2.2.x)需要企业信息作为输入
- **报告生成**类测试点(2.3.x)需要诊断结果作为输入

### 2. 功能依赖分析
- 查看测试描述中的"前置校验"和"准备"部分
- 识别需要"已完成"、"确保已有"等表述的测试点
- 分析测试流程中的逻辑顺序要求

### 3. 历史数据依赖分析
- **历史报告管理**(2.3.5a)需要多次报告生成来提供测试数据
- **报告对比**(2.3.5b)需要至少2份历史报告
- **趋势分析**类功能需要时间序列数据

### 4. 依赖关系类型
- **直接依赖**：A功能直接需要B功能的输出
- **数据依赖**：A测试需要B测试产生的数据
- **前置依赖**：A测试需要B测试验证的功能正常工作
- **批量依赖**：A测试需要多个测试点完成以获得足够数据

## 输出格式

```json
[
    {
        "description": "依赖关系的简要说明",
        "source": "前置测试点ID或ID数组",
        "target": "目标测试点ID"
    }
]
```

## 示例分析

基于测试计划中的描述：
- 如果某测试点描述提到"完成融资诊断后"，则它依赖于2.2.x系列测试点
- 如果某测试点描述提到"生成2-3份不同时间的诊断报告"，则它依赖于报告生成相关测试点
- 如果某测试点描述提到"确保已有至少2份不同时期的历史报告"，则它依赖于历史报告归档测试点

## 分析步骤

1. **读取测试计划**：仔细阅读每个测试点的description字段
2. **识别关键词**：查找"依赖于"、"需要"、"完成...后"、"确保已有"等关键表述
3. **分析测试类型**：根据metric名称判断测试点的功能分类
4. **建立依赖链**：构建从基础功能到高级功能的依赖链
5. **验证完整性**：确保所有依赖关系都有明确的source和target

请根据以上原则分析`detailed_test_plan.json`，生成完整的`test_dependencies.json`文件。
